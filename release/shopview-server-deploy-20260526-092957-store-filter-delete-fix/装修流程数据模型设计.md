# 装修流程数据模型设计

## 1. 文档目标

本文件基于 [装修流程阶段0详细设计.md](/Users/zhou/Library/Containers/com.tencent.xinWeChat/Data/Documents/xwechat_files/zhouzeze2011_6ba3/msg/file/2026-01/ShopView/装修流程阶段0详细设计.md)，输出阶段 1 可直接落库的数据模型设计。

目标如下：

- 明确装修流程需要新增哪些表
- 明确哪些现有表直接复用
- 明确表之间的主外键关系
- 明确阶段 1 必做字段、索引和约束
- 为后续 Alembic migration、SQLAlchemy model、接口设计提供依据

## 2. 设计原则

### 2.1 一条主线，一张主表

装修流程是一个贯穿申请、审批、施工、验收、结算、退款的主业务对象，因此必须有一张主表承载整个生命周期。

主表建议命名：

- `decoration_projects`

### 2.2 空间关联与流程实例拆开

装修涉及多个空间对象，审批又是另一套结构，因此不应把空间字段和审批节点全部堆进主表。

建议拆成三层：

- 主业务层：装修项目主表
- 业务明细层：装修项目空间、附件、节点结果
- 流程层：模板、实例、节点、动作

### 2.3 阶段 1 先做最小可闭环模型

阶段 1 不为了未来平台化一次性建太多表，只保留能支持主流程闭环的最小集合。

阶段 1 不建：

- 消息通知表
- 巡检记录表
- 整改闭环明细表
- 自由流或复杂 BPM 配置表

### 2.4 结构化优先，JSON 兜底

关键业务数据必须结构化落库，不能全部塞进 JSON。

可以放 JSON 的内容：

- 附件附加元数据
- 规则快照
- 审批动作附加字段
- 操作详情

必须结构化的内容：

- 项目状态
- 责任人
- 空间关联
- 审批节点状态
- 保证金、面积、电费、应退金额

## 3. 现有表复用策略

### 3.1 直接复用

以下表直接复用，不重复建：

- `users`
- `user_roles`
- `roles`
- `departments`
- `stores`
- `halls`
- `business_units`
- `operation_logs`

### 3.2 作为主数据关联使用

装修项目建议优先关联以下对象：

- 门店：`stores.store_id`
- 部门：`departments.id`
- 发起人/审批人/经办人：`users.user_id`
- 厅房：`halls.hall_id`
- 经营单元：`business_units.id`

### 3.3 不建议直接复用为主业务表

以下表不适合直接承担装修主流程：

- `contracts`
- `bills`
- `counters`
- `counter_groups`

原因：

- 它们是主数据或历史业务台账
- 不能承接装修流程的生命周期状态
- 不适合存审批和过程节点结果

## 6. 空间明细表设计

### 6.1 `decoration_project_spaces`

一个装修项目可能关联多个空间对象，因此需要明细表。

建议字段如下：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| id | BIGINT PK | Y | 主键 |
| project_id | BIGINT | Y | 所属装修项目 |
| space_type | VARCHAR(20) | Y | 空间类型：HALL/UNIT |
| hall_id | INTEGER | N | 厅房 ID |
| business_unit_id | BIGINT | N | 经营单元 ID |
| floor_id | BIGINT | N | 楼层 ID，冗余便于查询 |
| space_code | VARCHAR(100) | N | 空间编码冗余 |
| space_name | VARCHAR(200) | N | 空间名称冗余 |
| applied_area | NUMERIC(12,2) | N | 该空间申请面积 |
| measured_area | NUMERIC(12,2) | N | 该空间实测面积 |
| remarks | TEXT | N | 备注 |
| created_at | TIMESTAMP | Y | 创建时间 |
| updated_at | TIMESTAMP | Y | 更新时间 |

建议约束：

- `project_id` 外键到 `decoration_projects`
- `hall_id` 外键到 `halls`
- `business_unit_id` 外键到 `business_units`
- 校验规则：`space_type='HALL'` 时 `hall_id` 必填；`space_type='UNIT'` 时 `business_unit_id` 必填

建议索引：

- `ix_decoration_project_spaces_project_id`
- `ix_decoration_project_spaces_hall_id`
- `ix_decoration_project_spaces_business_unit_id`

设计说明：

- 主表保留项目级总面积
- 明细表保留每个空间的申请面积和实测面积
- 后续多空间结算时可以直接扩展，不需要改主表
