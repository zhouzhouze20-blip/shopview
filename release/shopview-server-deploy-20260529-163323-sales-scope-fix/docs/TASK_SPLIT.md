# ShopView Codex 任务拆分和提示词

## 1. 当前适合交给同事 Codex 的任务

| 任务 | 建议负责人 | 风险 | 说明 |
| --- | --- | --- | --- |
| 文档索引和启动说明维护 | 新同事 + Codex | 低 | 熟悉项目结构，不碰业务逻辑 |
| 前端单页面样式和交互优化 | 新同事 + Codex | 低到中 | 明确只改一个页面，避免全局组件 |
| Mock 数据补齐 | 新同事 + Codex | 低 | 适合无数据库环境演示 |
| 查询条件和表格字段调整 | 两边 Codex 配合 | 中 | 需要先确认接口字段和业务口径 |
| 权限配置页面完善 | 你主导，同事 Codex 辅助 | 中到高 | 涉及功能权限和数据范围 |
| 销售/合同/结算查询接口 | 你主导 | 高 | 涉及 ERP 数据口径和性能 |
| Docker 部署配置 | 你主导 | 中到高 | 涉及端口、环境变量、静态资源 |

## 3. 模块分工模板

### 前端页面任务

给 Codex 的提示：

```text
请只修改 [页面路径] 及必要的同模块 hooks/lib。
目标是：[具体页面行为或字段调整]。
保持现有 UI 风格，不要重构全局布局。
需要覆盖加载中、空数据、错误、无权限、正常数据状态。
完成后运行 cd client && npm run build。
```

适合文件：

- `client/src/pages/<module>.tsx`
- `client/src/hooks/use<Module>.ts`
- `client/src/lib/<module>.ts`

### 后端接口任务

给 Codex 的提示：

```text
请只修改 [router/model/schema 文件]。
目标 API：[路径和方法]。
查询表：[表名]。
请求参数：[path/query/body]。
返回字段：[字段和类型]。
权限要求：[登录/角色/数据范围]。
如果需要数据库迁移，先说明方案，等我确认后再改迁移文件。
```

适合文件：

- `python_app/routers/<module>.py`
- `python_app/models/<module>.py`
- `python_app/schemas/<module>.py`
- `python_app/alembic/versions/<revision>.py`

### 数据迁移任务

给 Codex 的提示：

```text
请先检查当前 alembic 版本和已有迁移，不要直接生成文件。
我要修改的目标表是：[表名]。
变更内容：[字段/索引/约束]。
请先给迁移方案、历史数据处理方式、验证 SQL 和回滚风险。
等我确认后再创建迁移文件。
```

注意：

- 同一时间只安排一个人负责迁移文件。
- 迁移文件提交前需要在开发库跑一遍。
- 涉及生产数据时先写备份和回退说明。

### 部署任务

给 Codex 的提示：

```text
请先只检查部署相关文件，不要修改。
目标环境是：[本地/测试服务器/生产服务器]。
问题是：[具体错误或目标]。
请列出涉及的 Dockerfile、compose、nginx、env 配置和端口影响。
确认方案后再做最小改动。
```

## 4. 建议先拆的三个小任务

### 任务 A：前端本地启动体验修正

目标：

- 统一前端开发时真实后端端口说明。
- 检查 `VITE_API_PROXY_TARGET`、`VITE_API_PORT` 的使用方式。
- 更新文档或脚本，让同事不用猜 7000 和 8000 的区别。

验收：

- 新同事按文档能打开 `http://localhost:5173`。
- 页面 API 请求能正常到达真实后端或 mock 后端。

提示词：

```text
请检查 ShopView 前端本地启动时 7000 和 8000 端口的配置差异，只改文档或启动脚本，不改业务页面。目标是让新同事按文档能选择真实后端 7000 或 mock 后端 8000。完成后说明改了哪些文件。
```

### 任务 B：选择一个低风险页面做字段补齐

候选页面：

- 门店：`client/src/pages/stores.tsx`
- 楼层：`client/src/pages/floors.tsx`
- 供应商：`client/src/pages/suppliers.tsx`

验收：

- 页面加载、空数据、错误状态完整。
- 对应接口字段在文档或任务里说明清楚。

提示词：

```text
请只处理 [页面文件] 的一个小改动：[具体字段/状态/交互]。先阅读该页面、相关 hook 和接口封装，再做最小修改。不要改全局组件。完成后运行 cd client && npm run build。
```

### 任务 C：整理一个业务专题文档

候选主题：

- 权限与数据范围。
- 装修流程。
- 活动分析。
- 销售分析口径。

验收：

- 文档有入口、背景、当前实现、待确认问题、下一步。
- `docs/README.md` 已补充链接。

提示词：

```text
请整理 [业务专题] 文档，只改 Markdown，不改代码。先阅读现有相关文档，然后输出背景、当前实现、待确认问题、下一步，并把入口补到 docs/README.md。
```

## 5. 每日同步建议

每天同步这四件事：

- 昨天完成了什么。
- 今天准备改哪些文件。
- 是否有阻塞。
- 是否需要别人先确认接口、数据或部署口径。

同步尽量落到任务或 PR 里，不只发在聊天窗口。
