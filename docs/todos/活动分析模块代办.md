# 活动分析模块代办

## 一期口径

- [x] 一期只分析卡券使用，不展开满减、折扣、赠品、银行卡等其他活动规则。
- [x] 暂不建设活动规则明细表，先以 `tktpopinfo` 活动档期和卡券相关字段为准。
- [x] 暂不建设活动归因关系表，先按卡券使用记录、小票记录和活动档期做分析。
- [x] 暂不建设活动成本表，成本和毛利直接取 `salegoodslist`。
- [x] 组织、柜组、部门维度直接使用现有组织柜组部门表。
- [x] 商品、品牌、品类维度直接使用现有商品相关表。
- [x] 会员信息取 `fj_dw_member_dim` 实时信息，暂不做会员分层快照。
- [x] 暂不建设活动前后对比汇总表，第一版先做活动/卡券结果分析。

## 一期重点校验

- [x] 校验卡券日志表 `tktcardfqlog` 中是否同时覆盖发券、用券、冲正/退券等业务动作。
- [x] 梳理 `tktcardfqlog.tcflzy` 的摘要取值，明确哪些代表发券、哪些代表用券、哪些代表冲正。
- [x] 梳理 `tktcardfqlog.tcflsource` 的券来源取值，确认销售返券、买券、退货返券、后台新增等分类。
- [x] 校验 `tktcardfqlog.tcflpopid` 是否能关联 `tktpopinfo.tpiid`：有关联的按活动档期归属；无关联的单独作为非档期发券/用券分析。
- [x] 校验 `tktcardfqlog.tcflsyjid + tcflinvno + tcflmkt` 是否能关联销售小票：可以关联销售小票。
- [x] 校验 `tktcardfqlog.tcflsyjtrace` 是否能辅助关联小票交易流水：只作为辅助排查字段，不参与主关联条件。
- [x] 校验 `salepay` 中 `paycode = '0500'` 和 `paycode = '0580'` 的付款记录是否都属于卡券使用：两类付款方式名称都是“卡券支付”。
- [ ] 校验 `salepay.payno`、`coupon_trace_seqno`、`coupon_event_id`、`coupon_policy_id` 与 `tktcardfqlog` 的对应关系。
- [x] 校验 `salehead.ybillno`、`salehead.ysyjh`、`salehead.yfphm` 是否能识别退货对应原小票：一期退货冲减先不做原单回溯，直接按 `salegoodslist` 中已计算字段汇总。

### 已确认付款口径

- `salepay.paycode = '0500'`：卡券支付。
- `salepay.paycode = '0580'`：卡券支付。
- 活动后台档期表使用 `tktpopinfo`。

### `tktcardfqlog.tcflsource` 来源字典

| 代码 | 含义 |
| --- | --- |
| `1` | 销售返券 |
| `2` | 前台买券 |
| `3` | 银行追送 |
| `4` | 退货返券 |
| `5` | 券转入 |
| `7` | 后台手工新增 |
| `8` | 后台买券 |

### 已确认金额正负口径

- `tcflzy = 'F'` 返券：按正数展示和汇总。
- `tcflzy = 'O'` 券消费：按正数展示和汇总。
- `tcflzy = 'U'` 券消费冲正：抵消券消费，按负数冲减净用券。
- 返券扣回、退货、冲正等其他动作：先按数据库 `tcflmoney` 原始正负汇总，不额外强制转正或转负。

### 已确认关联和经营口径

- 卡券日志关联销售小票的主条件：`tktcardfqlog.tcflmkt + tcflsyjid + tcflinvno` 对 `salehead.mkt + syjh + fphm`。
- `tktcardfqlog.tcflsyjtrace` 只作为辅助排查字段，不参与主关联条件。
- 卡券付款明细对账使用 `salepay.batch = tktcardfqlog.tcflsyjtrace` 精确匹配；否则同一小票上其他活动/非档期券付款会混入活动结果。
- 页面同时展示：
  - 卡券日志发生金额：`tktcardfqlog.tcflmoney`，包含发券、用券、扣回、冲正等日志动作。
  - 实际付款金额：`salepay.je`，仅包含纯付款。
- 经营结果以 `salepay.je` 为准。
- 对同一用券业务，卡券消费日志金额和卡券付款金额理论上应该一致；差异进入数据质量检查。

### 已确认活动归属口径

- `tktcardfqlog.tcflpopid = tktpopinfo.tpiid` 能关联时，归属到对应活动档期。
- 部分券为单独发券，可能不关联活动档期；这类券不强行归入活动档期，单独作为“非档期发券/用券”分析。
- `salepay` 有卡券付款但无法通过卡券日志归属活动时，进入“未归属卡券支付/数据质量”检查。

### 已确认退货冲减口径

- 一期退货冲减直接按 `salegoodslist` 中已计算字段汇总。
- 暂不通过 `salehead.ybillno / ysyjh / yfphm` 回溯原单归因。

### 已确认部门/柜组用券归属口径

- 一般卡券不会跨部门使用，通常是一张小票对应一个柜组。
- 特殊场景主要是超市部：可能一张小票跨柜组，且可能在超市的两个部门之间跨部门使用。
- 部门/柜组分析需要识别跨部门/跨柜组小票，避免把整张小票的卡券付款简单归到单一部门。

## 一期可直接产出的指标

- [ ] 活动档期清单：活动编码、活动主题、活动开始日期、活动结束日期。
- [ ] 卡券发生金额：按活动、日期、门店、会员、券号汇总 `tktcardfqlog.tcflmoney`。
- [ ] 卡券余额变化：按会员、券号、活动看 `tktcardfqlog.tcflye`。
- [ ] 卡券付款金额：按小票、活动、付款方式汇总 `salepay.je`。
- [ ] 0500/0580 付款金额：单独统计两类卡券付款方式。
- [ ] 活动销售额：关联小票后统计 `salehead` / `salegoods` / `salegoodslist` 销售金额。
- [ ] 活动成本和毛利：直接取 `salegoodslist` 中销售成本、毛利、净毛利相关字段。
- [ ] 部门/柜组贡献：通过销售表关联现有组织柜组部门表。
- [ ] 商品/品牌/品类贡献：通过销售表关联现有商品表。
- [ ] 会员贡献：通过会员号关联 `fj_dw_member_dim` 实时会员等级、注册渠道、状态等信息。
- [ ] 退货冲减：通过销售表原小票字段识别退货并冲减活动结果。

## 二期目标：活动设置与卡券结转闭环

### 背景问题

- [ ] 当前活动设置流程为业务部门先手工填写 Excel，营运部门汇总到总表后，再按 ERP 模块复制粘贴录入并审核，字段分散、校验弱、追溯难。
- [ ] 企划需要查看活动期内所有配置卡券，而不只是已经发生消费且能关联小票的几类卡券。
- [ ] 财务凭证侧目前只能看到部分 ERP 自动传递的用券借方、买券贷方；后台充值、银行打款、手工凭证无法稳定通过摘要匹配到具体券种、活动或批次。
- [ ] `tktqtype` / `tktqtypemkt` 中的折算收入比例会随活动变化，不能作为历史活动结转的实时取数依据，必须在活动配置时做快照。
- [ ] 部分后台充值券没有 `tcflpopid` 活动 ID，活动结束后发生退货时，需要回溯原销售小票和原券归属，不能按退货日期直接排除。

### 二期数据模型代办

- [ ] 建设活动主档表 `activity_config`：
  - 活动编码、活动名称、门店、活动开始/结束日期、活动状态、创建部门、营运确认状态、ERP 活动档期 ID。
  - 保留 ERP 回填字段：ERP 单据号、ERP 审核状态、审核日期、生效状态。
- [ ] 建设活动卡券配置表 `activity_coupon_config`：
  - 门店、活动、券字母、券名称、券描述、返券类别、券开始/结束日期、发放方式、投放渠道、投放数量。
  - 保存 `revenue_rate` 销售收入占比快照、`rate_source` 来源、`rate_confirmed_by`、`rate_confirmed_at`。
  - 保存面值、实收金额、升值金额；买券或升值券可按 `实收金额 / 券面值` 计算默认收入占比。
  - 同一券字母如果活动中途变更比例或有效期，不覆盖旧行，新增配置版本。
- [ ] 建设活动规则明细表：
  - 支持返券规则、用券规则、买券规则、满减规则、扣率设置、排除范围。
  - 记录适用范围：部门、柜组、供应商、品牌、商品、品类、楼层、门店。
  - 记录规则参数：门槛、满、送、用、减、限额、滚动方式、供应商承担比、门店承担比、活动时间。
- [ ] 建设活动卡券业务汇总台账：
  - 按 `activity_coupon_config` 汇总发放面值、核销面值、退货冲回、冲正、作废、过期、剩余面值、剩余销售收入。
  - 计算口径：`剩余面值 = 发放总额 - 净核销金额 - 作废/过期金额`；`剩余销售收入 = 剩余面值 * revenue_rate`。
- [ ] 建设财务凭证导入表 `finance_voucher_import`：
  - 存 ERP/财务系统拉取的凭证明细快照：凭证号、日期、摘要、科目、借方、贷方、制单人、来源系统、拉取批次。
- [ ] 建设卡券凭证关联表 `activity_coupon_voucher_match`：
  - 支持一张凭证拆分关联多个活动卡券配置，也支持多张凭证关联同一活动卡券配置。
  - 字段包括关联金额、借贷方向、匹配方式、匹配状态、确认人、确认时间、备注。

### 二期流程代办

- [ ] 将业务部门 Excel 填报改为柜位系统“活动申请明细”在线填报：
  - 按部门权限填活动、柜组、品牌、供应商、商品、活动类型、券种、规则、承担比例、时间和附件。
- [ ] 营运部门在系统内汇总和校验：
  - 校验柜组、商品、供应商是否存在。
  - 校验同一柜组/商品规则是否冲突。
  - 校验活动时间、券有效期、收入占比、供应商承担比是否缺失或异常。
  - 校验包含范围与排除范围是否冲突。
- [ ] 第一阶段先生成 ERP 录入/导入模板：
  - 输出活动档期主档、券种/付款方式设置、返券规则、用券规则、满减规则、扣率规则、排除规则、商品/柜组/供应商范围。
  - ERP 审核完成后回填 ERP 活动 ID、规则单号和审核状态。
- [ ] 第二阶段再评估调用 ERP 接口：
  - 柜位系统生成 ERP 活动规则草稿或凭证草稿。
  - ERP 返回单据号、凭证号和状态。
  - 正式凭证仍以 ERP/财务系统为准。

### 卡券结转与凭证关联代办

- [ ] 做“活动卡券结转”页面：
  - 按活动展示所有配置卡券，不只展示已消费并匹配小票的券。
  - 字段对齐现有结转 Excel：门店、活动名称、活动期、券有效期、券字母、券描述、返券类别、销售收入占比、发放总额、已核销面值、剩余金额、剩余销售收入、卡券状态、结转状态。
- [ ] 做“凭证关联”页面：
  - 左侧展示活动卡券结转行，右侧展示 ERP/财务凭证候选。
  - 支持按门店、日期范围、科目、借贷方向、金额范围筛选凭证。
  - 支持勾选凭证、填写本次关联金额、保存关联、取消关联。
- [ ] 做凭证差异检查：
  - 展示 `卡券剩余销售收入`、`已关联贷方金额 - 已关联借方金额`、`差异金额`。
  - 状态分为未关联、部分关联、已平、差异待处理。
  - 摘要关键字、日期和金额只能作为推荐匹配依据，最终以人工勾选确认为准。
- [ ] 后续可升级为“凭证申请单/凭证草稿”：
  - 柜位系统生成业务来源单据，审批后调用 ERP 接口生成待审核凭证。
  - ERP 回写凭证号、凭证日期、生成状态、请求报文和响应报文。

### 退货与无活动 ID 卡券归属口径

- [ ] 后台充值、银行打款、手工补录等无 `tcflpopid` 的卡券，通过 `activity_coupon_config` 归属，不强依赖 ERP 活动 ID。
- [ ] 活动结束后发生退货，应回溯原销售小票、原用券流水、原券号/券字母，冲减原活动卡券的净核销金额。
- [ ] 退货归属优先级：
  - 原小票关系：`salehead.ybillno / ysyjh / yfphm`。
  - 卡券流水关系：券号、会员号、`tcflsyjtrace`、`tcflzy` 退券/冲正动作。
  - 配置匹配：门店、券字母、券有效期、发放方式、批次或银行到账批次。
- [ ] 无法回溯原单的退货进入“退货待确认”异常池，不能直接丢弃，也不能默认归入退货当天的新活动。

### 销售收入占比快照口径

- [ ] 活动卡券配置建立时，从 `tktqtype` / `tktqtypemkt` 带出默认折算收入比例。
- [ ] 财务或营运确认后，将比例写入 `activity_coupon_config.revenue_rate` 快照。
- [ ] 历史分析、结转、凭证对账全部使用活动配置快照，不实时读取当前券种字典。
- [ ] 默认取值优先级：
  - 活动卡券配置手工确认值。
  - 门店券种配置。
  - 全局券种配置。
  - 面值和实收金额计算值。
  - 无法判断则标记为待确认，不自动结转。

### 卡券退损计算口径草案

- 卡券金额：以实际卡券付款金额 `salepay.je` 为主。
- 销售收入确认比例：可配置，例如 60%。
- 卡券退损金额：`卡券金额 * (1 - 销售收入确认比例)`。
- 供应商承担金额：直接取 `supgoodspayzkdet.sgpjglsupzk` 汇总。
- 退损后利润调整：`净毛利 + 供应商承担金额`。
- 示例：卡券 100 元，销售收入确认 60%，退损 40 元；供应商承担 50%，供应商承担金额 20 元，利润增加 20 元。
