# ShopView Codex 协作规范

## 1. 协作原则

- 代码统一通过 Git 管理，文档和代码一起提交。
- 一个人一个 Codex 会话，一个任务一个分支，避免两个 Codex 同时改同一批文件。
- 任务交给 Codex 前，先写清楚范围、验收和禁止改动项。
- 真实密钥、真实数据库密码、企业微信 Secret 不提交。
- Codex 的总结不能代替人工确认，合并前必须看 diff 和验证结果。

## 2. 推荐分支

| 分支 | 用途 |
| --- | --- |
| `main` | 稳定主分支，可部署版本 |
| `dev` | 日常集成分支，可选 |
| `feature/<模块>-<说明>` | 新功能开发 |
| `fix/<模块>-<说明>` | 问题修复 |
| `docs/<说明>` | 文档调整 |

示例：

```bash
git checkout main
git pull
git checkout -b feature/sales-dashboard-filter
```

## 3. 给 Codex 分任务的方式

每个任务至少写清楚：

- 背景：为什么要做。
- 范围：改哪些页面、接口、表或文档。
- 不做什么：本次明确不包含的内容。
- 验收：怎样算完成。
- 风险：是否涉及数据库、权限、部署或真实业务数据。
- 文件边界：本次优先允许改哪些目录，哪些目录不能碰。

推荐任务粒度：

- 一个前端页面或一个页面内的完整小功能。
- 一个后端接口组。
- 一个数据库迁移及其配套接口。
- 一个部署配置调整。
- 一个业务专题文档补齐。

不推荐给 Codex 的任务：

- “把系统优化一下”。
- “你看哪里有问题就改哪里”。
- “把销售模块全部做完”。
- “顺便重构一下”。

这些描述会导致改动范围失控，后续很难 review。

## 4. 前后端接口协作

接口变更先确认这些内容：

| 项目 | 说明 |
| --- | --- |
| URL | 例如 `/api/contracts/detail/{contract_no}` |
| 方法 | `GET`、`POST`、`PUT`、`DELETE` |
| 请求参数 | path、query、body 分开写 |
| 返回结构 | 列出关键字段、类型和空值规则 |
| 权限 | 是否需要登录、角色、数据范围 |
| 错误 | 常见错误码和错误信息 |

接口确定后：

- 后端方向的 Codex 先提供接口结构或 mock。
- 前端方向的 Codex 按接口封装在 `client/src/lib/` 或对应 `hooks/`。
- 联调问题记录到任务下，不在聊天里丢失。

## 5. PR / MR 规则

让 Codex 提交 PR 前，先要求它执行：

```bash
git status
cd client && npm run build
```

PR 描述包含：

- 改了什么。
- 怎么验证。
- 是否涉及数据库迁移。
- 是否涉及环境变量或部署配置。
- 截图或接口响应示例，适用于 UI 和接口改动。

Review 重点：

- 是否符合任务范围。
- 是否破坏已有页面和接口。
- 是否把密钥、临时文件、构建产物提交进来。
- 数据库迁移是否可重复执行、可回滚或有修复路径。
- 前端请求端口、路径和部署环境是否一致。

建议 review 提示词：

```text
请只做代码审查，不要修改文件。重点检查这次 diff 是否有 bug、越界改动、缺少验证、权限/数据口径/部署风险。按严重程度列出文件和行号。
```

## 6. 文件归属建议

| 范围 | 主要负责人 | 协作方式 |
| --- | --- | --- |
| `client/src/pages/` | 前端负责人 | 每人负责独立页面或模块 |
| `client/src/hooks/` | 前端负责人 | 与接口负责人约定返回结构 |
| `client/src/lib/api.ts` | 前后端共同确认 | 谨慎改动，影响全局请求 |
| `python_app/routers/` | 后端负责人 | 一个业务模块一个路由负责人 |
| `python_app/models/` | 后端负责人 | 涉及迁移时必须同步说明 |
| `python_app/alembic/versions/` | 后端负责人 | 不允许多人各自生成冲突迁移 |
| `Dockerfile*`, `docker-compose*.yml`, `config/` | 部署负责人 | 改动前先说明目标环境 |
| `docs/` 和根目录业务文档 | 任务负责人 | 功能完成时同步更新 |

## 7. 不建议的协作方式

- 两个人同时直接在 `main` 上开发。
- 只在聊天里描述接口，不写到文档或任务。
- 把本地 `.env`、压缩包、构建目录、上传文件提交到仓库。
- 一个 PR 混合页面重构、接口改动、数据库迁移和部署调整。
- 未确认真实数据口径就修改销售、合同、结算等核心查询逻辑。

## 8. 两个 Codex 同时开发的防冲突规则

- 不要让两个 Codex 同时改 `client/src/lib/api.ts`、`python_app/models/`、`python_app/alembic/versions/`、`Dockerfile*`、`docker-compose*.yml`。
- 每个人开始任务前先 `git status`，确认自己的分支和已有改动。
- 任务完成后先合并小 PR，再开始下一个大任务。
- 如果两个任务都要改同一个公共文件，先由一个人做接口或结构调整，另一个人等合并后再继续。
- Codex 遇到大量 unrelated diff 时，要求它“只基于当前任务改动，不要整理其他文件”。

## 9. 新同事第一周建议

第 1 天：

- 让 Codex 按 [开发接手指南](./DEVELOPMENT.md) 梳理项目并跑起来。
- 阅读 [系统架构梳理](./architecture/SYSTEM_ARCHITECTURE.md) 的第 1-4 节。

第 2-3 天：

- 负责一个低风险页面或文档任务，重点练习如何给 Codex 写清楚任务边界。
- 熟悉 `client/src/pages/`、`client/src/hooks/`、`python_app/routers/` 的对应关系。

第 4-5 天：

- 参与一个完整小功能，从接口到页面到 PR。
- 由熟悉项目的人 review 数据口径、权限和部署影响。
