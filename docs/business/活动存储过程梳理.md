# 活动库 GPP 存储过程梳理

范围：`建表/gpp/*.txt` 下当前版本的 13 个 Oracle PL/SQL package body。这里按“活动模块构成”和“后续柜位系统活动模块可复用/需注意点”整理。

## 总体模块划分

| 模块 | 包/文件 | 作用 |
| --- | --- | --- |
| 活动规则后台 | `POPMANA` / `Popmana.txt` | 审核、取消、发布组合促销、规则促销、数量/印花/价随量变促销，把后台单据展开成前台可用的 `Popcurinfo`、`Poprule`、时间段、门店维度规则。 |
| 销售满送/返券计算 | `salepopms` / `salepopms.txt`，`SALEFLMS`，`SALEPOPMS_GIFT` | 从销售小票 `sellhead/selldetail/sellpayment` 读取促销规则，计算返券、满赠、银行追送、退货扣回、换货处理，并写券余额、券日志和分摊表。 |
| 券生命周期管理 | `Ticketmana`，`Tktfqline`，`TICKETBATCH`，`tktparty` | 后台券单据审核、券批次调整、手工券发放/激活、第三方券销售/冲正/查询/激活、券延期/转移/手工加券等。 |
| POS/外部接口 | `Gppposinterface`，`Poptktex`，`popinterface`，`ws_interface_pop` | 前台查券、用券、买券/退券、网联接口、折扣规则查询、WebService 买券等。 |
| 停车券/停车结算 | `Tkpmanager` | 停车规则、停车券单据、停车券结算及供应商/合同分摊。这个包体现了活动费用如何落到供应商/合同/分摊对象，对柜位活动结算口径有参考价值。 |

## 各包与过程

### 1. `POPMANA`：活动规则后台发布中心

核心职责：把后台维护的活动单据审核成前台查询/计算能用的规则数据。它是“活动配置 -> 可执行规则”的主入口。

| 过程 | 用途 |
| --- | --- |
| `Qtypedelcheck` | 券种删除前校验，检查券种是否已被活动/规则引用。 |
| `Approve` | 统一审核入口，按单据类别分派到组合促销、规则促销、印花/数量促销、价随量变等审核逻辑。 |
| `Bapprove_Combpop` | 审核组合促销单，校验门店、时间、状态，将明细写入当前促销规则。 |
| `Bapprove_Pm` | 审核价随量变类促销。 |
| `Cmdcancel_Combpop` / `Cmdcancel_pmpop` | 取消已发布到命令/门店的组合促销、价随量变促销。 |
| `Auditpoprule` / `Cancelpoprule` | 审核/取消满减、满赠、满额、包装等基础促销规则。 |
| `Auditpoprule_q` / `Cancelpoprule_q` | 券相关促销规则的审核/取消。 |
| `p_Autocreatecode`、`p_01`、`p_02` | 规则自动编码和按规则类型生成/处理的辅助入口。 |
| `Sendpopgift` / `Cancelpopgift` | 满赠赠品发放/取消。 |
| `Approve_Rule` | 规则促销单据审核入口，校验 `Prcpophead/Prcpopdetail` 和 `Popgift`。 |
| `Submit_Rule` | 把规则促销按商品、系列、门店、时间条件展开到 `Popcurinfo`。 |
| `Insert_Ruletime` / `Insert_Resulttime` / `Insert_Timefb` | 处理活动时间段、跨天时间段、结果规则时间。 |
| `Insert_Popcurinfo` | 写入前台当前促销信息。 |
| `Submit_Pctj` | 处理“部分满足/规则条件”类赠送规则。 |
| `Updateflag` | 更新单据状态。 |
| `Cancel` | 后台活动单据取消入口。 |
| `Approve_Sl` | 数量、印花类促销审核。 |
| `Submit_Curpop` | 把普通促销明细转成当前促销执行记录。 |
| `Checkbilldetail` | 审核前明细校验，补默认值，校验商品、折扣、金额、条件规则。 |

关键表：`Prcpophead`、`Prcpopdetail`、`Popgift`、`Popgiftrule*`、`Popcurinfo`、`Poprule`、`Popruledet`、`Billmktdetail`、`Goodsbase`。

### 2. `salepopms`：销售返券/满送主计算引擎

核心职责：POS 销售入库后，按小票和商品行匹配活动，生成券、满赠、追送，并处理退货扣回。

| 过程 | 用途 |
| --- | --- |
| `gettktpopinfo` | 根据档期取券有效期、券生成方式、使用范围、是否固定面值、打印地点。 |
| `refreshfqrule` | 后台重置商品返券规则，用于前台找规则失败后的补偿。 |
| `uptshfqflag` | 更新销售小票返券/追送处理状态。 |
| `appendsh` | 多张小票合并到同一个事务。 |
| `sp_salemsrule` | 单张小票匹配返券/满赠规则，按商品行计算有效返券金额，写 `tktsellfqdetail`。 |
| `upttkt` | 写入或更新券余额 `tktcardfqtotal`，并写券流水 `tktcardfqlog`。 |
| `calctkt_sa` / `calctkt_fq` / `calctkt` | 按规则明细生成不同类型的券金额和分摊。 |
| `sp_salefq` | 销售返券核心入口，按事务汇总规则，生成券余额、券日志和 `tktsellfqamt` 分摊。 |
| `sp_salesa_scrule` / `sp_salesa_sc` / `sp_salesa` | 商场/银行追送规则计算。 |
| `b_salems` | 满送计算总控入口：清理旧中间表、逐票匹配规则、生成返券、追送、日汇总、状态更新。 |
| `salems` | 对外简化入口，调用 `b_salems`。 |
| `Mzrulecale` | 满赠规则计算入口。 |
| `changeqno` | 修改券号关联。 |
| `th_rollback` | 退货/冲正时回滚原用券流水，恢复券余额。 |
| `salethhctkt` | 按券批次扣回退货应扣的返券。 |
| `tktseq` | 记录/生成扣回券序列。 |
| `uptdaytotal` | 更新券日汇总。 |
| `salethhc` | 退货、红冲、换货的扣回总入口。 |
| `htthcancel` / `htthsendok` | 后台退货取消/确认后的返券处理。 |
| `salehh` | 换货场景的返券与扣回处理。 |
| `jobrun` / `job_submit` / `job_remove` | 定时任务入口。 |

关键表：`sellhead`、`selldetail`、`sellpayment`、`selllisttemp`、`tktsellfqdetail`、`tktsellfqamt`、`tktsellsadetail`、`tktcardfqtotal`、`tktcardfqlog`、`tktrule`、`tktrulefqdet*`、`poprule`、`mzgiftselldetail*`。

### 3. `SALEFLMS` / `SALEPOPMS_GIFT`：满赠和赠品库存分支

这两个包和 `salepopms` 的部分逻辑高度相似，偏“返礼/满赠/赠品库存”。`SALEFLMS` 有更完整的满赠生成、取消、退回；`SALEPOPMS_GIFT` 更像抽出来的赠品处理版本。

| 过程 | 用途 |
| --- | --- |
| `Sp_Salemsrule` | 小票级满赠/返礼规则匹配。 |
| `Mzrulecale` | 满赠计算入口。 |
| `Appendtranslist` | 创建参与满赠计算的事务小票列表。 |
| `Genmzgift` | 根据满赠规则生成赠品明细，涉及赠品库存。 |
| `Cancelmzgift` / `Returnmzgift` | 取消或退回满赠赠品。 |
| `Calmzlist` | 计算满赠清单、整单/单件限额、已赠数量。 |
| `Approve` / `Bapprove` | 赠品换货单审核。 |

## 活动模块主链路

1. 后台维护活动：`POPMANA` 审核活动单据，生成 `Popcurinfo`、`Poprule`、时间段、门店范围。
2. POS 销售前/销售中查规则：`popinterface.sp_findzksarule`、前台规则查询逻辑读取 `Popcurinfo` 等当前规则。
3. 销售完成后算返券/满赠：`salepopms.salems -> b_salems -> sp_salemsrule -> sp_salefq/sp_salesa/Mzrulecale`。
4. 生成券资产：统一落到 `tktcardfqtotal`，所有变动落到 `tktcardfqlog`。
5. POS 用券/查券：`Gppposinterface.Sp_Findtkt/Sp_Sendtkt` 或网联版本。
6. 退货/红冲/换货：`salepopms.salethhc/salethhctkt/th_rollback` 和 `Gppposinterface` 的扣回逻辑。
7. 后台管理：`Ticketmana`、`Tktfqline`、`TICKETBATCH`、`Poptktex` 做券批次、手工券、买券、延期、转移、作废等管理。

## 对柜位系统活动模块的启示

当前 GPP 活动库不是简单“活动表”，而是一套“单据审核 -> 规则发布 -> 前台执行 -> 资产/分摊/日志 -> 退货回滚”的闭环。柜位系统如果直接做活动模块，建议不要只做活动配置页面，要同步考虑这些边界：

| 活动模块问题 | 需要从现有系统确认 |
| --- | --- |
| 活动适用范围 | 需要确认活动按门店、柜组、商品、品牌、品类、供应商、会员、银行卡等哪些维度生效。现有 GPP 同时支持商品、柜组、品类、品牌、会员卡、银行卡。 |
| 活动类型边界 | 至少要区分折扣、满减、返券、满赠、买券、追送、停车券、印花/数量促销，以及哪些先做、哪些互斥、哪些可叠加。 |
| 规则发布方式 | 现有活动模块是后台审核后生成 `Popcurinfo` 等当前规则，并通过 `Cmdsend` 下发门店。柜位系统要决定是实时查询、审核发布，还是生成前台缓存。 |
| 销售计算时点 | 需要确认活动是在销售前查询价格/折扣，还是销售后按小票生成返券/赠品；GPP 两类都有。 |
| 分摊和承担口径 | 促销让利、券分摊、满减折扣、赠品成本由商场、供应商、柜组、品牌或合同方谁承担。这里会影响结算单和报表。 |
| 券资产管理 | 如果做返券/买券/手工券，需要券余额表、券流水表、券有效期、券状态、券号、券种、冲正和延期，而不是只存活动结果。 |
| 退货/红冲/换货 | 需要完整扣回逻辑：原销售返过什么券、用过什么券、退货时扣多少、券余额不足怎么处理。 |
| 审核和追溯 | 活动规则需要草稿、审核、取消、终止、生效期、版本和操作日志，否则后面很难解释历史销售为什么按某个活动算。 |

## 还需要你补充的资料

1. 你希望柜位系统先做哪些活动类型：折扣、满减、返券、满赠、买券、停车券、银行卡活动、会员活动，最好先定一期范围。
2. 活动适用对象口径：门店、楼层、柜组、商品、品牌、品类、供应商、会员等级、银行卡号段哪些必须支持。
3. 现有前台/POS 调用清单或 package spec：现在只有 package body，能看实现，但不能百分百区分哪些过程是外部正式 API。
4. 生产库中脱敏样例：5-10 条活动主档、活动明细、销售小票、券余额、券流水、退货小票、结算/分摊明细。
5. 活动费用承担口径：商场承担、供应商承担、柜组承担、品牌承担，是否要进入供应商结算或合同结算。
6. 柜位系统目标边界：只做活动分析/查询，还是要做活动配置、审核发布、POS 执行、返券满赠、退货扣回。
