# ShopView 企业微信接入规划

## 1. 目标

本次企业微信接入拆成两个登录入口和一个组织同步能力：

1. 浏览器扫码登录：用户在 PC 浏览器打开 ShopView，登录页展示企业微信二维码，员工用企业微信 App 扫码确认后进入系统。
2. 企业微信内部应用：用户在企业微信工作台打开 ShopView 应用，自动完成企业微信网页授权并进入系统。
3. 通讯录同步：同步企业微信部门、成员、成员离职/禁用状态，并转换为 ShopView 的用户、部门归属和业务数据权限基础。

实施优先级：先做浏览器扫码登录，再做企业微信内部应用，最后完善通讯录变更回调和数据权限自动映射。

## 2. 现有系统基础

当前项目已经具备企业微信接入的大部分基础表和权限链路：

- `users` 是系统唯一用户主体。
- `user_identities` 已预留 `identity_type = password / wecom`，并包含 `corp_id`、`wecom_user_id`、`union_id`。
- `departments`、`posts`、`user_department_posts` 已能表达用户的门店、部门、岗位归属。
- `roles`、`permissions`、`user_roles` 已承载功能权限。
- `data_policies`、`data_policy_items` 已承载统一业务数据范围，当前 `business_scope.view` 已用于合同、销售、结算等模块。
- 现有 `/api/auth/login` 只支持用户名密码，后续企业微信登录应复用现有 Cookie JWT 登录态。

结论：企业微信不要另建一套账号和权限体系，应接入到现有 `users + user_identities + user_roles + data_policies`。

## 3. 企业微信侧准备

企业微信管理员需要准备：

1. 企业 ID `CorpID`。
2. 自建应用：
   - `AgentId`
   - 应用 `Secret`
   - 应用可见范围，第一版建议只放测试部门。
3. 通讯录同步权限：
   - 通讯录 `Secret`，或具备读取通讯录能力的应用 Secret。
   - 能读取部门列表、成员列表、成员详情。
4. 授权回调域名：
   - 例如正式系统为 `https://shopview.example.com`。
   - 企业微信后台通常配置域名 `shopview.example.com`，不带路径。
5. HTTPS 公网地址：
   - 浏览器扫码登录和企业微信回调都需要公网可访问的 HTTPS 地址。
   - 本地调试可临时使用 HTTPS 内网穿透，但正式环境不能依赖内网穿透。
6. 如果要实时感知员工离职、部门调整，需要配置通讯录回调：
   - 回调 URL，例如 `https://shopview.example.com/api/wecom/callback`
   - `Token`
   - `EncodingAESKey`

## 4. 配置项

建议新增后端环境变量：

```env
WECOM_ENABLED=true
WECOM_CORP_ID=
WECOM_AGENT_ID=
WECOM_APP_SECRET=
WECOM_CONTACT_SECRET=
WECOM_REDIRECT_BASE_URL=https://shopview.example.com
WECOM_CALLBACK_TOKEN=
WECOM_CALLBACK_AES_KEY=
WECOM_AUTO_CREATE_USER=false
```

说明：

- `WECOM_APP_SECRET` 用于扫码登录、企业微信内部网页授权获取用户身份。
- `WECOM_CONTACT_SECRET` 用于同步组织通讯录。若企业微信后台权限模型允许复用应用 Secret，可以先不单独配置。
- `WECOM_AUTO_CREATE_USER=false` 是第一版推荐策略：企业微信用户没有同步/绑定到 ShopView 时拒绝登录，避免所有可见员工自动进入系统。

## 5. P0 浏览器扫码登录

### 5.1 用户流程

1. 用户打开 ShopView 登录页。
2. 点击“企业微信扫码登录”。
3. 前端请求后端生成扫码登录参数。
4. 登录页渲染企业微信二维码。
5. 用户用企业微信 App 扫码并确认。
6. 企业微信回调 ShopView：`/api/auth/wecom/callback?code=...&state=...`。
7. 后端用 `code` 换取企业微信用户身份。
8. 后端用 `corp_id + wecom_user_id` 查询 `user_identities`。
9. 找到本地用户后，写入现有 `shopview_auth_token` Cookie。
10. 重定向回系统首页。

### 5.2 后端接口

建议新增：

```text
GET /api/auth/wecom/login-url
GET /api/auth/wecom/callback
```

`GET /api/auth/wecom/login-url` 返回：

```json
{
  "login_url": "企业微信扫码授权地址",
  "state": "一次性state"
}
```

`GET /api/auth/wecom/callback` 处理：

1. 校验 `state` 防 CSRF 和重放。
2. 获取企业微信 `access_token`。
3. 用 `code` 获取访问用户身份。
4. 根据 `corp_id + UserId` 查 `user_identities`。
5. 校验用户 `is_active=true` 且 `status` 不为 `DISABLED / LOCKED`。
6. 写登录日志，`identity_type = wecom`。
7. 写 Cookie JWT。
8. 跳转到前端首页或登录前目标页。

### 5.3 前端改造

修改登录页：

- 保留账号密码登录。
- 新增“企业微信扫码登录”按钮。
- 点击后嵌入二维码，或跳转到企业微信扫码授权页。
- 扫码回调成功后由后端重定向回系统。

前端不保存企业微信 Secret，不直接请求企业微信服务端 API。

### 5.4 绑定策略

第一版推荐：

- 只允许已同步/已绑定的企业微信用户登录。
- 找不到 `user_identities(identity_type='wecom', corp_id, wecom_user_id)` 时，提示“账号未开通，请联系管理员”。

后续可选：

- 自动创建用户，但默认无角色、无数据权限。
- 管理员在系统配置页补角色和数据范围。

## 6. P1 企业微信内部应用

### 6.1 用户流程

1. 用户在企业微信工作台打开 ShopView 自建应用。
2. 前端或后端发现当前入口是企业微信内访问。
3. 跳转企业微信网页授权地址。
4. 企业微信回调 `/api/auth/wecom/callback`，带 `code` 和 `state`。
5. 后端复用 P0 的 callback 逻辑，换取企业微信用户身份并写入 ShopView Cookie。
6. 用户进入系统。

### 6.2 与浏览器扫码登录的关系

两种登录最终都进入同一个后端回调和同一套用户绑定逻辑：

```text
企业微信扫码登录
企业微信内部应用网页授权
        ↓
/api/auth/wecom/callback
        ↓
user_identities
        ↓
users / roles / data_policies
        ↓
shopview_auth_token
```

区别只在授权入口 URL 和前端触发方式，不应拆成两套账号处理逻辑。

## 7. P1 通讯录同步

### 7.1 同步对象

同步企业微信：

- 部门列表
- 成员列表
- 成员详情
- 成员启用/禁用/离职状态
- 成员部门归属
- 手机号、邮箱、姓名、员工编号等基础字段

落地到 ShopView：

- `users`
- `user_identities`
- `departments`
- `user_department_posts`

### 7.2 同步策略

第一版建议做后台手动同步接口：

```text
POST /api/wecom/sync/departments
POST /api/wecom/sync/users
POST /api/wecom/sync/all
```

后续再补：

```text
POST /api/wecom/callback
```

用于接收企业微信通讯录变更回调。

### 7.3 离职员工处理

不建议物理删除 `users`。

企业微信成员离职或禁用时：

```text
users.status = DISABLED
users.is_active = false
```

同时：

- 保留历史登录日志和操作日志。
- 保留历史业务单据关联。
- 登录时拒绝该用户。
- 可选择停用其 `user_department_posts`。
- 可选择停用或删除企业微信来源的数据权限策略。

## 8. 部门与数据权限映射

需要先明确一个关键口径：企业微信部门是否等于 ShopView 的经营部门。

如果完全一致：

- 企业微信部门 ID 或部门名称可以映射到 `departments.dept_code / dept_name`。
- 用户部门归属可以直接转成 `user_department_posts`。
- 可以自动生成 `data_policies` 的 `department` 维度。

如果不完全一致，推荐新增映射表：

```text
wecom_department_mappings
- id
- corp_id
- wecom_department_id
- wecom_department_name
- store_id
- shopview_department_id
- data_scope_dimension_type
- data_scope_dimension_value
- is_active
- created_at
- updated_at
```

这样可以表达：

- 一个企业微信行政部门映射到一个经营部门。
- 一个企业微信部门映射到多个柜组。
- 一个企业微信部门只作为登录组织，不参与数据权限。

第一版建议不要自动把所有企业微信部门直接变成业务数据权限，先走映射表或管理员确认。

## 9. 建议新增模块

后端：

```text
python_app/services/wecom_client.py
python_app/services/wecom_sync.py
python_app/routers/wecom.py
```

职责：

- `wecom_client.py`：封装企业微信 access_token、扫码/网页授权用户身份、部门、成员 API。
- `wecom_sync.py`：把企业微信部门和成员转换为本地用户、身份、部门归属。
- `wecom.py`：暴露同步接口、通讯录回调接口。

认证相关可扩展：

```text
python_app/routers/auth.py
```

新增扫码登录 URL 和 callback。也可以拆成 `routers/wecom_auth.py`，但最终仍复用现有 auth Cookie。

前端：

```text
client/src/pages/login.tsx
client/src/contexts/AuthContext.tsx
```

登录页新增企业微信扫码登录入口，`AuthContext` 可增加 `loginWithWecom()` 或只让按钮跳转后端生成的登录 URL。

## 10. 数据库调整建议

已有表可用，但为了可运维，建议新增或补充字段：

### 10.1 departments 补来源字段

```sql
ALTER TABLE departments
  ADD COLUMN IF NOT EXISTS source_type VARCHAR(20) DEFAULT 'MANUAL',
  ADD COLUMN IF NOT EXISTS source_system VARCHAR(50) DEFAULT 'shopview',
  ADD COLUMN IF NOT EXISTS external_dept_id VARCHAR(100),
  ADD COLUMN IF NOT EXISTS synced_at TIMESTAMP;
```

### 10.2 user_identities 已基本够用

建议确认唯一索引：

```text
corp_id + wecom_user_id 唯一
identity_type + identifier 唯一
```

### 10.3 可选新增同步日志

```text
wecom_sync_logs
- id
- sync_type
- status
- started_at
- finished_at
- created_count
- updated_count
- disabled_count
- error_message
```

## 11. 安全要求

1. 企业微信 Secret 只放后端环境变量。
2. 前端不得暴露 `Secret`。
3. 登录 callback 必须校验 `state`。
4. `state` 应有短有效期，并在使用后失效。
5. 企业微信登录成功后仍要检查本地用户状态和权限。
6. Cookie 生产环境应启用 `secure=true`。
7. 所有企业微信登录写 `login_logs`，`identity_type=wecom`。
8. 通讯录回调要校验企业微信签名，并解密消息后处理。
9. 对企业微信 API 做超时、重试和错误日志。

## 12. 里程碑

### M1 浏览器扫码登录

- 配置企业微信自建应用、授权回调域名。
- 新增企业微信登录 URL 接口。
- 新增企业微信登录 callback。
- 登录页新增企业微信扫码登录按钮。
- 已绑定企业微信用户可扫码登录。
- 未绑定用户拒绝登录并提示联系管理员。
- 登录日志可看到 `identity_type=wecom`。

验收标准：

- PC 浏览器打开 ShopView 可扫码登录。
- 停用用户扫码后不能登录。
- 刷新页面后仍能通过现有 `/api/auth/me` 获取登录用户。

### M2 通讯录手动同步

- 新增企业微信部门同步。
- 新增企业微信成员同步。
- 自动写入 `user_identities(identity_type='wecom')`。
- 离职/禁用员工自动停用本地用户。
- 系统配置页可看到同步来的用户和部门。

验收标准：

- 企业微信测试部门成员能同步到 ShopView。
- 企业微信禁用/移除成员后，ShopView 用户被停用。
- 同步不会覆盖管理员手工配置的角色和数据权限。

### M3 企业微信内部应用

- 企业微信工作台配置 ShopView 应用入口。
- 企业微信内打开应用可自动授权登录。
- 复用 M1 的 callback 和用户绑定逻辑。

验收标准：

- 员工从企业微信工作台进入 ShopView，无需再次输入账号密码。
- 非可见范围或未绑定员工不能进入。

### M4 通讯录回调和数据权限自动映射

- 接入通讯录变更回调。
- 新增或完善企业微信部门到 ShopView 经营部门/柜组的映射。
- 可按映射自动生成或更新 `business_scope.view`。

验收标准：

- 企业微信部门/成员变更后，ShopView 能自动同步状态。
- 数据权限变更可追踪来源，不覆盖手工配置。

## 13. 待确认问题

1. 企业微信部门是否与 ShopView 经营部门一致？
2. 是否需要一个企业微信部门映射多个经营部门或柜组？
3. 员工编号以企业微信 `userid`、手机号，还是 ERP 工号为准？
4. 企业微信同步是否允许自动创建用户？
5. 自动创建用户时默认赋予哪些角色？建议默认无业务权限。
6. 离职员工是否只停用，还是同时清理企业微信来源的数据权限？
7. 正式域名和 HTTPS 证书是否已准备？
8. 第一批试点部门和测试账号是谁？
