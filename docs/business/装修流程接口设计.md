## 2. 设计原则

### 2.1 接口分层

阶段 1 接口分为四类：

- 项目主数据接口
- 附件接口
- 流程/待办接口
- 节点动作接口

### 2.2 REST 为主，动作接口为辅

对象查询与维护使用 REST 风格。

流程流转使用动作接口，例如：

- `submit`
- `approve`
- `reject`
- `return`
- `confirm`

### 2.3 阶段 1 不暴露复杂流程编排接口

阶段 1 不做：

- 自由流程设计接口
- 动态加签接口
- 任意节点跳转接口
- 通用 BPM 引擎开放接口

### 2.4 权限在后端控制

前端可以隐藏按钮，但后端仍必须校验：

- 当前用户是否具备功能权限
- 当前用户是否是当前节点责任人或具备管理员权限
- 当前状态是否允许执行该动作

## 5. 项目主数据接口

### 5.1 `GET /api/decorations`

用途：

- 装修项目列表页
- 门店查询
- 状态筛选
- 角色视角查询

查询参数建议：

| 参数 | 必填 | 说明 |
| --- | --- | --- |
| `store_id` | N | 门店筛选 |
| `status` | N | 状态筛选 |
| `keyword` | N | 按项目编号、品牌、施工单位模糊搜索 |
| `mine` | N | 是否仅看我发起的 |
| `assignee_me` | N | 是否仅看当前由我处理 |
| `planned_entry_date_from` | N | 计划进场开始 |
| `planned_entry_date_to` | N | 计划进场结束 |
| `skip` | N | 偏移 |
| `limit` | N | 数量 |

返回结构建议：

```json
[
  {
    "id": 1,
    "project_no": "DEC20260424001",
    "store_id": 601,
    "store_name": "总部店",
```

### 5.4 `PUT /api/decorations/{project_id}`

用途：

- 编辑草稿
- 被退回后的资料补充

限制：

- 仅 `DRAFT` 状态允许完整编辑
- 非 `DRAFT` 状态只允许按节点开放有限编辑，不通过这个接口做

请求体结构与创建类似。

### 5.5 `POST /api/decorations/{project_id}/submit`

用途：

- 发起进场审批

请求体建议：

```json
{
  "comment": "资料已提交，请审核"
}
```

后端动作：

1. 校验草稿必填字段
2. 校验必需附件
3. 创建或启动 `ENTRY` 流程实例
4. 项目状态改为 `PENDING_PROPERTY_REVIEW`
5. 生成首个待办节点

返回建议：

```json
{
  "message": "提交成功",
  "project_id": 1,
  "current_status": "PENDING_PROPERTY_REVIEW",
  "current_node_code": "PROPERTY_REVIEW"
}
```

### 5.6 `POST /api/decorations/{project_id}/cancel`

用途：

- 取消项目

限制：

- 仅发起人或管理员可取消
- 已完成项目不可取消

请求体建议：

```json
{
  "reason": "业务撤销申请"
}
```

## 6. 附件接口

### 6.1 `GET /api/decorations/{project_id}/attachments`

用途：

- 单独拉取附件列表

查询参数：

- `attachment_type`
- `related_node_code`

### 6.2 `POST /api/decorations/{project_id}/attachments`

用途：

- 绑定上传后的文件到装修项目

说明：

- 文件本体仍复用现有对象上传接口
- 本接口只负责写 `decoration_attachments`

请求体建议：

```json
{
  "attachment_type": "DRAWING_DRAFT",
  "file_name": "1F装修图纸.pdf",
  "file_url": "/uploads/xxx.pdf",
  "file_size": 123456,
  "mime_type": "application/pdf",
  "related_node_code": "PROPERTY_REVIEW"
}
```

### 6.3 `DELETE /api/decorations/{project_id}/attachments/{attachment_id}`

用途：

- 删除附件关联

限制：

- 草稿状态可删除
- 非草稿状态仅允许删除当前退回节点新增附件，阶段 1 可先限制为草稿可删

## 8. 节点动作接口

阶段 1 不做统一通用动作接口，直接按业务节点拆接口，便于前后端理解和实现。

### 8.1 `POST /api/decorations/{project_id}/property-review`

用途：

- 物业审图

请求体：

```json
{
  "review_result": "APPROVED",
  "review_comment": "图纸符合要求",
  "rectification_requirements": null
}
```

允许结果：

- `APPROVED`
- `RETURNED`

后端动作：

1. 校验当前状态必须为 `PENDING_PROPERTY_REVIEW`
2. 校验当前用户为物业责任人
3. 写入 `decoration_property_reviews`
4. 写入 `workflow_actions`
5. 更新 `workflow_instance_nodes`
6. 流转项目状态

### 8.2 `POST /api/decorations/{project_id}/leader-approval`

用途：

- 领导审批进场

请求体：

```json
{
  "action": "APPROVE",
  "comment": "同意进场"
}
```

允许动作：

- `APPROVE`
- `REJECT`

### 8.3 `POST /api/decorations/{project_id}/deposit-confirm`

## 10. 错误处理约定

建议统一返回：

```json
{
  "detail": "错误说明"
}
```

常见错误码：

- `400` 参数错误或状态不允许
- `401` 未登录
- `403` 无权限
- `404` 项目不存在
- `409` 状态冲突
- `500` 系统异常

## 11. 权限校验建议

### 11.1 列表与详情

- 先校验登录
- 再按角色与数据范围过滤

### 11.2 动作接口

必须同时校验：

1. 项目存在
2. 当前状态匹配
3. 当前用户具备对应角色权限
4. 当前用户是该节点责任人或管理员

### 11.3 可执行动作返回

详情接口建议返回 `available_actions`，由后端统一计算。

这样前端不需要自己推断状态和权限。
